Customizing your installation

There are a few steps that you MUST do when starting up a Rakontu site. After the software is installed,

  1. Change the SITE_SUPPORT_EMAIL address in site_configuration.py. This MUST be the email address associated with the Google account of one of the administrators of the application. If it isn't, people in Rakontus on your site won't be able to send each other emails. If you don't want to list your personal email here, create a Google account with a separate email you use only for this purpose.

Backup and restore using AppRocket

AppRocket is an open source library that backs up and restores a Google App Engine datastore to/from a MySQL database. Using AppRocket you can keep a copy of your Rakontu database on your local machine in case of any problems, or you can do stuff with the data in SQL.

To backup using AppRocket, do these things:

  1. Install a MySQL server on your local computer. Set up a user with a password for AppRocket to use.
  2. Edit the file config.py on your Rakontu GAE site so that it has the correct entries in the Replication URL and MYSQL DATABASE CONFIGURATION sections of the file. 
    # Replication URL - change this to URL corresponding your application
ROCKET_URL = "http://www.yoururl.com/rocket"
...
# MYSQL DATABASE CONFIGURATION
DATABASE_HOST = "localhost"
DATABASE_NAME = "rakontu_or_whatever_you_want_to_call_it"
DATABASE_USER = "approcket_or_whatever_you_want_to_call_it"
DATABASE_PASSWORD = "whatever_you_set_the_approcket_password_to"
  3. Make sure that "/rocket" comes after your Rakontu site url in the ROCKET_URL constant.
  4. Set the CURRENT_MODE constant in config.py to one of the modes listed in the file. (If you just want to backup, keep it as RECEIVE.)
  5. Make sure you have Python 2.5 or greater installed on your system. (To find out what you have installed, type "python -V" at a command prompt.)
  6. Copy rocket/station.py from your GAE installation to a local machine.
  7. Start your MySQL server. Create an empty database called rakontu (or whatever you named it in your config.py).
  8. Run station.py with the PythonLauncher or within IDLE (these come with the Python distribution).

To restore using AppRocket: Presumably you already did all the stuff in the backup section. Just edit rocket/config.py on your GAE installation and change the mode to one that sends, then run station.py.

Notes on using AppRocket:

  • Two classes of object in the GAE datastore are not backed up to MySQL: Helps and Exports. Helps come from the system and can't be changed per community, and are identical to what is in the "help.csv" file, so there is not much reason to back them up. Exports are only kept as temporary generations for download and can easily be created again. Also, some will be large enough to make MySQL choke. So these two classes are not downloaded.
  • If you have multiple Rakontus on your site, they will be mixed together in the MySQL database. Since every major data item that can possibly be part of a Rakontu has a "rakontu" reference field, you can use filters to select only those items in any one Rakontu. The table field will be called "rakontu_ref". However, sadly, AppRocket (as of this writing) doesn't deal well with list fields in GAE objects, so all list-field items will be in separate tables, linked to their parent object by a "k" field. So you have to retrieve the data for one Rakontu in two steps: first, get all parent objects by the "rakontu_ref" field; then second, retrieve their child objects by their "k" fields. Messy but doable.
  • If you need to make any sort of bulk change to Rakontu objects, you should be able to set the AppRocket mode to RECEIVE, make changes in the updated MySQL database, set the AppRocket mode to SEND, then update the online database. However, take care to do this at some time when your users are not likely to be using the sytem (or temporarily disconnect it to make the change).
  • The AppRocket backup includes inactivated Rakontus, members, questions, and characters. (Because so many things point to these objects, they are never removed, only made invisible. Entries, annotations, answers and links are removed, because otherwise the site could get bloated with "dead wood".) If for some reason you don't want anyone to see inactivated items, be aware that they will be in the MySQL database. You can remove them by doing a query there on the "active" field and removing those with FALSE values.
